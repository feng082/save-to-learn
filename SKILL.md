# Save to Learn

## 目标

把值得学习的内容变成可追溯、可分析、可检索、可复用的素材资产。默认使用用户当前指定的学习素材库；若当前目录存在 `inbox/`、`sources/`、`reports/`、`index.md` 和 `rules.md`，将其视为素材库根目录。

## 命令与触发

支持以下显式命令，也支持自然语言触发：

- `/save-to-learn`：自动判断素材类型并完成收录、分析、报告和索引。
- `save` 或 `/save-to-learn save`：完成本地收录后，同步更新 HTML 展示页，并提交、推送到 GitHub。
- `/save-to-learn inbox`：处理 `inbox/` 中尚未处理的素材。
- `/save-to-learn search <关键词>`：按标题、类型、标签、STYLE 或风格检索。
- `/save-to-learn style <STYLE编号>`：读取指定 STYLE 的风格总结和相关报告，给出复用方案。
- `/save-to-learn find-style <关键词>`：寻找适合指定用途的图片风格。
- `/save-to-learn index`：检查并更新索引链接和记录。

自然语言示例：`收录这个链接`、`收录这个本地视频`、`分析并收录这张图片`、`处理 inbox`、`找适合 AI 教程的信息图风格`、`参考 STYLE-001`。

## 素材库结构

只使用以下基础结构，不擅自增加分类层级、单素材文件夹、数据库、向量库、RAG、网页前端、大量 JSON 或复杂依赖：

```text
学习素材库/
├─ inbox/
├─ sources/
├─ reports/
├─ index.md
└─ rules.md
```

若用户明确要求新建素材库，只创建上述基础结构和规则/索引模板；不安装依赖、不下载网络内容、不创建演示素材或脚本，除非用户另行同意。

## 处理流程

1. 判断素材类型：图片、文字/PDF、视频/音频、网页链接、GitHub/工具或本地文件。
2. 去重检查：比较原始 URL、本地绝对路径、文件哈希、标题和近似图片；完全重复不新建素材或报告。
3. 保存原始入口：网络使用 `.url`，本地使用绝对路径 `.lnk`，无法创建快捷方式时使用 `.md` 引用文件。
4. 生成 `reports/` 中对应的 Markdown 研读报告。
5. 图片分配或复用 STYLE 编号，必要时更新对应风格总结。
6. 将记录放入 `index.md` 表格顶部，并确认报告链接和原始入口有效。
7. 处理 `inbox/` 时，只有确认已完成处理的临时文件才清理。
8. 汇报新增内容、报告、STYLE、重复项和无法访问/分析的内容。
9. 当用户明确说 `save` 时，完成“本地收录 → HTML 更新 → GitHub 提交推送”全流程；仅说“分析”“看看”或“本地保存”时，不自动发布到 GitHub。

## save 的 HTML 与 GitHub 同步流程

本学习素材库已配置公开展示页。用户说 `save` 时，按以下顺序执行，不需要再次询问是否同步：

1. 先完成本地 `sources/`、`reports/`、`index.md` 的去重、分析和收录。
2. 更新本地素材库根目录的 `index.html`：在 `materials` 数组中新增或修正对应记录，确保报告路径、标题、类型、来源、标签和推荐程度一致。
3. 同步公开站点工作副本 `E:\资料库\05_图片视频\自媒体\学习素材库\github-site`：更新其中的 `index.html` 和对应公开报告；本地绝对路径、`.lnk` 快捷方式和大型本地原文件不得上传，改为“本地素材未公开”或只保留原始网络链接。
4. 校验公开页面没有失效的 `file:///` 链接、没有本机盘符路径，报告文件存在，HTML 结构完整。
5. 在 `github-site` 中执行 `git diff --check`，只提交本次素材同步相关文件；确认没有混入其他未授权改动后，提交并推送到 `feng082/save-to-learn` 的 `main` 分支。
6. 汇报本地收录、HTML 更新、GitHub 提交哈希、推送结果和线上页面地址。GitHub 或网络失败时，本地收录仍保留，但必须明确标记“线上同步失败”，不能声称已发布。

固定路径与发布约定：

- 本地展示页：素材库根目录 `index.html`
- GitHub 工作副本：`E:\资料库\05_图片视频\自媒体\学习素材库\github-site`
- GitHub 仓库：`https://github.com/feng082/save-to-learn`
- GitHub Pages：`https://feng082.github.io/save-to-learn/`
- 只有用户明确说“只保存本地”“不要上传”或“不要提交 GitHub”时，才跳过第 3–6 步。

## 网页读取失败时的 Firecrawl 二次尝试

对网页、文章、帖子、GitHub 页面和视频页面，必须采用以下顺序：

1. 先用当前环境正常打开并读取原始页面。
2. 如果页面只有空壳 HTML、依赖 JavaScript、正文为空、内容加载不完整或当前读取方式失败，再尝试 Firecrawl。
3. 先检查 `FIRECRAWL_API_KEY` 环境变量；有 Key 时优先使用 Firecrawl Hosted API，默认地址为 `https://api.firecrawl.dev`。
4. 没有可用 API Key 时，检查本地源码目录 `C:\Users\10167\.codex\tools\firecrawl` 和本地服务 `http://localhost:3002`。
5. 如果官方源码目录不存在，先执行浅克隆：

```powershell
New-Item -ItemType Directory -Force -Path 'C:\Users\10167\.codex\tools' | Out-Null
git clone --depth 1 https://github.com/firecrawl/firecrawl.git 'C:\Users\10167\.codex\tools\firecrawl'
```

6. 源码下载成功不等于 Firecrawl 服务已经运行。没有 API Key 且 `localhost:3002` 不可用时，不要假装抓取成功；说明“Firecrawl 源码已下载，但服务尚未启动”，然后按无法访问处理。
7. 不要擅自执行 Docker Compose、安装 Rust/Redis/PostgreSQL/pnpm 或启动长期后台服务，除非用户明确要求部署 Firecrawl。
8. Firecrawl 成功时，优先取得 Markdown，并根据已读取内容生成报告；不要把网页全文复制到 `sources/`。
9. 对 JavaScript 动态页面，可使用等待渲染或浏览器交互；对登录墙、验证码、私密内容、删除页面或强反爬页面，失败后仍标记“链接无法访问”。
10. 如果普通读取和 Firecrawl 都失败，只使用已经确认的标题、简介、作者或封面信息，绝不虚构正文、时间轴或分析结论。

Firecrawl 是备用读取通道，不改变原始链接保存规则。无论哪种方式读取成功，`sources/` 仍只保存原始 `.url`，报告中记录实际使用的读取方式和分析状态。

## 原始入口规则

- 网络页面、文章、帖子、公众号、知乎、X、GitHub、视频页面和在线图片页面：只保存原始页面 `.url`，不复制全文，不擅自下载完整视频或网页全部图片。
- `.url` 格式：

```ini
[InternetShortcut]
URL=https://example.com
```

- 本地文本、图片、PDF、音频和视频：不复制、移动或修改原文件；用 PowerShell 与 `WScript.Shell` 创建指向绝对路径的 `.lnk`。失败时写 `.md`，记录原路径、文件名、类型、大小和最后修改时间。
- 用户明确允许保留的小型图片可以直接放入 `sources/`；大型文件默认只保存快捷方式。

## 命名规则

统一使用：`YYYY-MM-DD_素材类型_简短中文标题`。

- 标题使用有意义的简短中文，文件名尽量不超过 40 个汉字。
- 禁止 `image1`、`video2`、`abc`、`未命名` 等无意义名称；删除 Windows 非法字符。
- 同名追加 `_02`、`_03`，不覆盖已有文件。
- 报告和 Sources 入口使用相同基础名称。

## 报告规则

每份报告开头必须包含：

```markdown
# 素材标题

- 素材类型：
- 来源平台：
- 作者或发布者：
- 发布时间：
- 收录时间：
- 原始链接：
- 本地原始路径：
- Sources入口：
- 标签：
- 推荐程度：1–5星
- 分析状态：完整 / 部分 / 无法访问
```

未知信息填写“未知”，不猜测。

### 文字、网页和 PDF

分析一句话概括、摘要、3–10 个核心观点、论证结构、值得学习的具体原因、值得质疑的地方、可复用写作方法、抽象内容结构、延伸选题、实际应用和留空的“我的想法”。

### 视频和音频

分析一句话概括、可确认的时间/章节结构、开头钩子、叙事节奏、画面设计、声音设计、可能的传播原因、制作方法、脚本结构、改进点、延伸选题和实际应用。无法读取时不虚构内容或时间轴。

### GitHub 项目和工具

分析解决的问题、核心功能、适用人群、使用门槛、安装运行方式、技术栈、维护状态、优缺点、同类区别、实际价值和明确的使用建议。

## 图片与 STYLE 规则

每张图片都建立视觉风格档案，分析用途、第一眼吸引力、版式架构、画风、色彩、文字层级、图标/装饰、制作方式、亮点、缺点和 5–12 条可执行设计规则，并生成风格参考提示词。

每张图片必须有：

- 1 个相对稳定的主风格标签
- 3–8 个副标签
- 1 个稳定的 `STYLE-xxx` 编号

相近图片复用已有 STYLE；只有整体视觉体系明显不同才新建编号。相同 STYLE 下有至少 2 张图片时，只维护一份 `reports/STYLE-xxx_风格总结.md`。

图片含大量准确文字、严格网格、精确对齐、统一卡片或规则模块时，优先建议 HTML/CSS、PPT 或 Figma 排版，再配合 AI 生成插画，不要一律判断为 AI 直接生图。提示词复用视觉规律，不复制品牌、Logo、水印、签名、原文案或独特受保护元素。

以后收到图片生成或风格参考请求时，先查 `index.md`，再读取风格总结和最相关的 1–3 份单图报告，最后组织本次内容的版式、提示词和制作方案。除非用户明确说“生成图片/开始生图/直接做图”，默认不调用生图工具。

## 索引与真实性边界

`index.md` 使用以下表格，新增记录放在顶部：

```markdown
| 收录日期 | 类型 | 标题 | STYLE编号 | 主标签 | 副标签 | 来源 | 推荐 | 报告 | 原始素材 |
|---|---|---|---|---|---|---|---|---|---|
```

索引只负责导航；普通素材的 STYLE 和主标签填写“—”，普通标签控制在 3–8 个。

- 无法访问链接时明确写“链接无法访问”。
- 只能读取标题、简介或封面时，只根据已确认信息分析。
- 不虚构作者、日期、视频内容、时间轴、字体名称或精确颜色；HEX 只能作为近似值。
- 少量引用原文，以总结、拆解和学习为主。
- 报告明确区分客观信息、原作者观点、分析判断和对用户的可执行建议。

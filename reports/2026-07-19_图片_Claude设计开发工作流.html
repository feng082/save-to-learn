一句话结论

这张图把“设计—实现—测试—优化”的 AI 开发闭环压缩成一张可快速复习的长图：先让设计交付结构化代码，再让 Claude Opus 以小版本构建 MVP，随后用真实运行和反馈驱动修复，最后才进入性能优化与功能扩展。

内容摘要

图中分为四段：第一段强调 Claude Design 输出 HTML、CSS、React 和 data.js；第二段强调以 UI-only 的最小版本开始，并明确技术栈；第三段要求每个版本都能运行，通过测试反馈区分代码问题和 UI 问题；第四段建议先分析、重构和优化性能，再继续添加功能。底部还记录了把 Claude Design 从 Web 体验转成本地 Skill 的过程。

可复用要点

  • 设计交付优先提供结构化代码和数据，而不是只给截图。
  • MVP 先做主屏或单一核心路径，并提前声明技术栈。
  • 每次迭代都必须可运行，让人和 Agent 都能获得即时反馈。
  • 用 git diff、截图和实际运行结果把问题分成代码问题与 UI 问题。
  • 性能问题先定位和修复，再进入下一轮功能扩展。

视觉拆解

版式结构

纵向长图由顶部标题区、四个编号分区和底部横向收束流程组成。每段遵循“左侧解释文字—中部流程示意—右侧判断或结果”的阅读路径，适合手机端从上到下浏览。

画风与色彩

米白纸张底、黑色手写体、蓝色主标题，辅以黄色荧光笔、绿色成功状态、橙色问题提示和紫色对比框。线条保留轻微手绘感，但编号、箭头和分栏仍然清晰，形成“亲切教程感 + 工程秩序感”。

文字层级与图标

英文主标题最大，章节标题次之，正文用短句和项目符号降低阅读负担。手机、代码文件、Git、机器人、文件夹和折线图等简化线稿承担快速识别功能。

可执行设计规则

  1. 用一条横向标题带交代主题、受众和结果。
  2. 主体控制在 3–5 个编号章节，每章只讲一个阶段。
  3. 每章固定使用“原则 + 简短说明 + 流程图/对比图”。
  4. 米白纸张底配深色手写字，降低技术内容的生硬感。
  5. 蓝色作为主色，黄/绿/橙/紫分别承担强调、成功、问题、比较语义。
  6. 用粗线框、虚线框和箭头区分容器、判断和时间顺序。
  7. 复杂句压缩成 1–2 行，关键动作使用动词开头的短句。
  8. 流程末尾给出明确的“结果已验证”状态。
